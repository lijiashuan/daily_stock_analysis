# TSV 格式解析问题修复

## 🐛 问题描述

**症状：**
```
Parsed DataFrame columns: ['成交日期\t成交时间\t证券代码\t...']
DataFrame shape: (169, 1)
Skip reasons summary: {'missing_trade_date_column': 169}
```

**根本原因：**
- 文件是 **TSV 格式**（制表符分隔）
- 但被当作 CSV 解析时，没有指定制表符分隔符
- 导致整行被当作一个字段，所有列都在一列中
- 系统找不到"成交日期"列（因为它实际上是整个字符串的一部分）

## ✅ 解决方案

### 修复内容

修改了 `_read_csv_robust()` 方法，**优先尝试 TSV 格式**：

```python
# Strategy 1: Try TSV (tab-separated) - most common for broker exports
for encoding in ("utf-8-sig", "gbk", "gb18030"):
    df = pd.read_csv(
        io.BytesIO(content),
        encoding=encoding,
        sep='\t',  # ← 关键：指定制表符分隔符
        dtype=str,
        keep_default_na=False,
        engine='python',
    )
    if not df.empty and len(df.columns) > 1:
        return df  # 成功解析为多列
```

### 解析策略顺序

1. **TSV (制表符分隔)** ← 新增，优先尝试
2. **CSV (逗号分隔)**
3. **TSV (容错模式)**
4. **自动检测分隔符**
5. **默认设置**

### 验证条件

每种策略都会验证：
- ✅ DataFrame 非空
- ✅ **列数 > 1**（确保不是单列数据）

## 🎯 适用场景

这个修复特别适用于：

1. **券商导出的 TXT 文件**
   - 华泰证券、中信证券等
   - 通常使用制表符分隔
   - 扩展名可能是 .txt 或 .xls

2. **Excel 另存为的文本文件**
   - 选择"文本文件(制表符分隔)"
   - 实际是 TSV 格式

3. **数据库导出文件**
   - 很多系统默认使用 TSV 格式
   - 因为制表符不太可能出现在数据中

## 🔍 如何确认文件格式

### 方法 1：用文本编辑器查看

用 Notepad++ 或 VS Code 打开文件：
- 如果看到列之间有明显的空白（制表符），就是 TSV
- 如果看到逗号分隔，就是 CSV

### 方法 2：查看日志

修复后的日志会显示：
```
Successfully parsed as TSV with gbk encoding, 169 rows, 14 columns
```

这表明文件被正确解析为 TSV 格式，有 14 列。

### 方法 3：检查列数

- **TSV/CSV 正确解析**：列数 > 1（如 14 列）
- **解析失败**：列数 = 1（所有数据在一列中）

### 对于自制文件

1. **推荐使用 TSV**：

   - 不易与数据内容冲突

2. **或使用 CSV**：
   - 逗号分隔
   - 更通用
   - 注意处理包含逗号的字段

## 📝 示例

### 正确的 TSV 格式

```txt
成交日期	成交时间	证券代码	证券名称	业务名称	成交价格	成交数量
20240101	09:00:00	600519	贵州茅台	买入	1800.00	100
20240102	10:00:00	600520	五粮液	卖出	200.00	50
```

**注意**：列之间是**制表符**（TAB），不是空格。

### 错误的格式

```txt
成交日期 成交时间 证券代码  # 使用空格分隔，可能解析失败
20240101 09:00:00 600519
```
